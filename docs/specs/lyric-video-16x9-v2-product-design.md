# Lyric Video 16:9 v2 Product Design

更新时间：2026-06-07

## 1. 标题与状态

- 标题：`lyric-video-16x9-v2` 官方音乐作品感视频模板设计
- 作者：Codex
- 状态：Design baseline for implementation
- 适用范围：真实 AI 出歌后的 16:9 MP4 成片、Claude Web v1 成品预览、公司社区发布素材交接
- 上游依据：PRD v0.3、技术方案 v0.2、`docs/specs/cover-video-rendering-v0.1.md`、公网真实完整体验 smoke 结果

## 2. 设计目标

`lyric-video-16x9-v2` 的目标不是做游戏宣传片，也不是做花哨的活动页视频，而是让每个玩家生成的歌曲看起来像一支可被官方社区接收的音乐作品：

1. 第一眼是“燕云十六声相关音乐作品”，不是通用古风 PPT。
2. 画面安静、克制、有质感，能长时间播放 2-4 分钟不疲劳。
3. 歌词可读性优先，字幕必须稳定烧录在安全区内。
4. 音频、封面、歌词、波形、进度和结尾信息形成完整成片，不再只是静态封面配字。
5. 模板可配置、可复用、可降级，为后续 9:16、活动包装和更高级字幕对齐预留空间。

## 3. 当前问题

当前 `LyricVideo16x9` 只是工程样例，已能证明 Remotion/FFmpeg 工具链可跑，但不满足真实产品体验：

- 画面结构偏左右分栏样例，不像正式音乐作品。
- 封面没有真正作为视觉主资产进入模板。
- 歌词承载区域像 UI 卡片，缺少 MV 感。
- 没有片头、片尾、进度、波形和官方作品识别。
- 当前真实样本导出的 MP4 没有音频轨道；这是 P0 交付缺口，必须在模板实现时一并修复。

## 4. 最终方向

采用“专辑封面式歌词 MV”作为默认模板。

画面不是传统游戏 HUD，也不是卷轴书法展示，而是类似音乐平台官方曲目视觉化：封面作为核心视觉，背景由封面虚化和轻微电影化光影组成；歌词以清晰下三分之一字幕承载；底部有低调波形和播放进度；片头片尾用燕云乐坊识别和作品信息完成收束。

参考方向只借鉴结构和动效，不直接拷贝开源模板资产：

- `nexu-io/html-video` 的 `frame-light-leak-cinema`：借鉴轻电影光漏、颗粒、暗场层次。
- Remotion audiogram 模板方向：借鉴封面、音频、字幕、波形组合的结构。
- `reactvideoeditor/remotion-templates` 中的 lower third、text highlight、film burn、noise grain、Ken Burns、end card 组件方向：借鉴可复用动效模块。

## 5. 画面结构

默认画面为 1920×1080，30fps。

### 5.1 背景层

- 使用生成封面作为全屏背景源。
- 背景 `object-fit: cover`，放大 108%-116%，做 18-32px 模糊。
- 叠加暗场渐变，保证歌词区和标题区可读。
- 叠加极轻颗粒、暗角和低透明度光漏；不能出现廉价霓虹、重紫蓝渐变或大面积装饰光球。
- 背景有缓慢 Ken Burns 推拉，整首歌幅度不超过 4%，避免眩晕。

### 5.2 封面主体层

- 主封面以“专辑封面板”形式展示，位置默认在画面左中或居中偏左。
- 封面板建议尺寸 560-640px，圆角 0-8px，阴影克制。
- 当前 Image2 封面可能是 16:9；模板应支持 16:9 原图背景，同时对专辑封面板做中心安全裁切。后续封面 Prompt 应要求主体居中、四周留安全边距，便于方形或近方形裁切。
- 封面板不能被字幕遮挡；字幕和封面之间保留明确视觉层级。

### 5.3 文案信息层

- 片头显示歌名、燕云乐坊标识、用户昵称或 Mock 用户名、作品编号。
- 主体阶段保留歌名小字或简短元信息，但不长期占据视觉中心。
- 片尾回收歌名、燕云乐坊、作品编号，可加“玩家原创曲”一类普通用户能理解的文字。
- 视频内不出现“发布包 JSON”“object key”“provider”等技术词。

### 5.4 歌词字幕层

- 字幕在下三分之一安全区，首期句级字幕。
- 每屏 1-2 行，长句自动断行；单行建议不超过 18-22 个中文字符。
- 字号建议 54-72px，标题式歌词可短暂放大，但不能挡住封面主体。
- 使用浅米白文字、细描边或柔和阴影，保证亮/暗背景均可读。
- 当前行突出，上一句/下一句可选低透明度辅助，但不能形成 K 歌 UI。
- 字幕淡入淡出 8-12 帧；不做夸张弹跳、旋转或频繁位移。

### 5.5 音频可视化层

- 开启可视化辅助，但保持低调。
- 默认用底部细波形或频谱条，宽度约 50%-70%，高度 36-72px。
- 波形颜色使用暗金、米白或低饱和青绿，透明度随音量变化。
- 首版如果暂不做真实音频分析，可使用确定性伪波形；但真实成片验收应优先升级为基于音频数据的波形。
- 可视化不能抢歌词层级，不能做夜店式频闪。

### 5.6 进度层

- 底部增加极细播放进度线。
- 进度线只表达作品完成度，不模拟播放器控件。
- 进度线需要在成品视频中自然存在，不应让用户误以为可交互。

## 6. 时间结构

### 6.1 片头：0-2.5 秒

- 背景从封面模糊场缓慢显现。
- 封面板进入，歌名和燕云乐坊标识出现。
- 可以有一次轻微金色印章式闪现，但不能做复杂古风卷轴展开。

### 6.2 主歌/副歌主体：2.5 秒至倒数 3 秒

- 封面板持续存在，背景缓慢运动。
- 歌词按句级 timeline 轮换。
- 波形和进度持续工作。
- 副歌或情绪较强段落可以轻微提高光漏、颗粒和背景对比，但不改变整体模板结构。

### 6.3 片尾：最后 3 秒

- 歌词淡出，歌名回收。
- 展示“燕云乐坊 / 玩家原创曲 / 作品编号”。
- 背景回到封面主视觉。
- 最后一帧适合成为视频缩略图，不出现空白或纯黑。

## 7. 风格边界

应该像：

- 官方音乐作品视觉化。
- 专辑封面动态版。
- 克制的游戏社区发布素材。
- 有燕云气质，但不过度古装化。

不应该像：

- 古风 PPT 模板。
- 满屏卷轴、印章、毛笔字、祥云边框。
- 游戏战斗 HUD。
- 酒吧频谱视频。
- AI 生成图直接铺满后随便加字。

## 8. 工程落点

### 8.1 模板与组件

继续使用 Remotion，不引入 `html-video` 作为生产渲染依赖。新增或重构为以下边界：

- `LyricVideo16x9V2`：默认 production template。
- `TitleCard`：片头歌名与燕云乐坊识别。
- `AlbumCoverStage`：封面主体和背景运动。
- `LyricLowerThird`：安全区歌词字幕。
- `AudioVisualizer`：波形或频谱。
- `ProgressRail`：视频进度线。
- `OutroCard`：片尾作品信息。

### 8.2 渲染输入

`RenderWorkerJobInput` 需要在现有字段基础上补足可被 Remotion 直接读取的素材来源：

- 音频来源：必须能被 `<Audio>` 读取。`audio_object_key` 不足以直接渲染，需要由 Java local-process 边界提供临时音频文件路径、可读 URL，或 render-worker 内部具备对象存储读取能力。
- 封面来源：同理，`cover_object_key` 需要解析为可读图片文件或 URL。
- 模板版本：输出 metadata 应记录 `template_id=lyric-video-16x9-v2`。
- 安全区：输出 timeline JSON 记录字幕安全区和模板版本。

### 8.3 音频修复

当前真实样本 MP4 没有声音，v2 实现必须修复：

- Remotion composition 中加入 `<Audio>`。
- `renderMedia` 不得继续使用 `muted: true`。
- 输出 MP4 必须包含音频轨道，目标编码为 AAC。
- 渲染后用 `ffprobe` 校验 video stream 和 audio stream 同时存在。
- 视频时长与音频时长差异不超过 1000ms；超出时标记为视频渲染失败，不进入可交接成品。

### 8.4 质量校验

v2 模板完成后，新增或强化以下检查：

- render-worker 单元测试覆盖动态时长、空歌词、长歌词断行、模板 metadata。
- 本地 `render:job` smoke 输出含音频轨道的 MP4。
- 公网真实完整体验 smoke 导出的成片必须通过 `ffprobe` 音视频双轨检查。
- 抽样检查 390px Web 成品页视频预览可播放；视频本身仍按 16:9 输出。

## 9. 降级策略

- 封面生成失败：使用默认燕云主题封面，仍走 v2 模板。
- 字幕 timeline 失败：切换 `cover-video-16x9-v1` 或 v2 的 no-lyrics 模式，只保留封面、歌名、波形和进度。
- 音频不可读：不得生成无声 MP4；作品应进入可解释失败，并允许重试音乐或视频渲染。
- 波形生成失败：可降级为确定性伪波形，不阻断成片。
- 片头/片尾元信息缺失：使用 Mock 用户名和作品编号兜底。

## 10. 验收标准

| 项目 | 标准 |
|---|---|
| 画面质感 | 看起来像正式音乐作品，不像工程样例或 PPT |
| 音频 | MP4 包含可解码音频轨道，非静音输出 |
| 字幕 | 句级字幕在安全区内，长句不溢出，不遮挡封面主体 |
| 封面 | 作为背景和主体双重使用，不明显拉伸、不严重裁切 |
| 时长 | 视频覆盖完整音频，音视频时长差异不超过 1000ms |
| 可视化 | 底部波形/频谱存在但不喧宾夺主 |
| 片头片尾 | 歌名、燕云乐坊和作品编号出现，最后一帧可作为缩略图 |
| 降级 | 封面、字幕、波形失败均有明确兜底；音频失败不得产出假成功 |
| 证据 | 真实完整体验样本能导出音频、封面、视频和 timeline，并通过脱敏记录留证 |

## 11. 非目标

- 本批不做 9:16 竖版视频。
- 本批不做逐字 K 歌高亮。
- 本批不做复杂 3D 场景、剧情镜头或角色动画。
- 本批不接公司发布、分享、审核真实系统。
- 本批不把 `html-video` 替换为生产渲染引擎。

## 12. 下一步实施顺序

1. 修复 MP4 无音频问题，先让现有模板输出有声视频。
2. 实现 `LyricVideo16x9V2` 的基础结构：背景、封面板、字幕、进度和片尾。
3. 加入低调可视化辅助，先伪波形，后续可升级真实音频分析。
4. 增加 ffprobe 音视频双轨校验。
5. 复跑真实完整体验 smoke，导出一支可供用户观看的有声 v2 成片。
